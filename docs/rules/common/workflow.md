# 작업 프로세스

## 의도 파악

- 사용자의 힌트에서 의도를 추론한다
- 추론한 내용을 명시적으로 정리하고 확인 요청 후 진행
- 반만 이해하고 작업하지 않는다

### 예시
```
사용자: "Storage가 모호해... __native는 순수 네이티브만..."
→ 추론: Storage 삭제 의도, __native 순수화, 점진적 마이그레이션
→ 확인: "이 이해가 맞나요?"
```

## 질문 방식

### 옵션 제시 전 5가지 검증
1. **의도 파악**: 현재 구조/리팩토링의 목적이 무엇인가?
2. **대상 구분**: 적용 대상들이 본질적으로 같은 것인가?
3. **원칙 검증**: 이 원칙이 왜 좋은가? 이 상황에서도 유효한가?
4. **반대 검증**: 적용 안 하면 어떤 문제가 생기는가?
5. **구조 검토**: 현재 제약을 받아들여야 하는가? 구조 개선이 더 나은가?

### 옵션 등급
| 등급 | 설명 |
|------|------|
| **최고의 제안** | 장기적 관점에서 최선 |
| **최적** | 현재 상황에 가장 적합 |
| **일반** | 표준적/보편적 방식 |
| **고려하지 않음** | 제외 이유 명시 |

## 정합성 검증

중요 결정(아키텍처, 패턴 변경 등) 전 최소 3회 확인:
1. **1차**: 요구사항 파악
2. **2차**: 이해한 내용 요약 및 확인
3. **3차**: 구체적 구현 방향 확인

## 플래닝

### 핵심 원칙
모든 코딩 작업 전 현재 파이프라인 레벨에 맞는 **계획 → 확인 → 실행** 순서를 따른다.
승인 필요 여부와 등급은 `_domains/_agent-skills/supervisor/PIPELINE-DECISION.md`의 L0/L1/L2/L3 기준을 따른다.
플랜 작성 완료 ≠ 승인. L2/L3에서는 사용자의 명시적 승인 응답을 확인한 뒤 Developer를 실행한다.

### 플랜 저장
- Supervisor 작업(L0~L3)의 실행 보고서는 `_domains/{domain}/reports/YYYYMMDD/작업명.md`에 작성한다
- `{domain}`은 `_domains/README.md`의 도메인 맵으로 결정한다
- 크로스 도메인 작업은 주 도메인 보고서에 작성하고, 영향 도메인을 본문에 함께 기록한다
- 날짜 폴더가 없으면 생성한다
- 작업 중 플랜이 변경되면 같은 파일을 업데이트한다
- 실행 착수 전 보고서가 없으면 생성하고, 있으면 최신 계획으로 업데이트한다
- 상세 생성/갱신 시점은 `_domains/_agent-skills/supervisor/EXECUTION-MANUAL.md`의 "파이프라인 실행 보고서" 절차를 따른다

### 규칙 참조 프로세스 (5단계)
| 단계 | 행동 |
|------|------|
| 플랜 작성 전 | rules/ 전체 확인했음을 명시 |
| 플랜 작성 후 | 참고한 규칙 명시 |
| 코딩 전 | 적용할 규칙 명시 |
| 코딩 후 | 반영한 규칙 명시 |
| 작업 완료 후 | 일반화할 수 있는 원칙 의견 요청 |

### 룰 감사 게이트

개발 태스크는 코딩 전에 적용 대상 rule file 목록을 실제 파일 시스템에서 산출하고, 필요한 파일을 끝까지 읽은 뒤 실행 보고서에 기록한다.

필수 절차:

1. `rg --files`로 적용 대상 rule file 목록을 산출한다.
2. `.claude/rules/` 공통/코딩 룰과 작업 대상 도메인의 `_domains/{domain}/CLAUDE.md`, `_domains/{domain}/rules/*.md`를 감사 대상에 포함한다.
3. 다른 도메인의 runtime, config, scaffold, API, DB, SP 계약을 참고하면 해당 도메인도 `_domains/{domain}/CLAUDE.md`, `_domains/{domain}/rules/*.md` 기준으로 감사 대상에 포함한다.
4. 각 rule file을 끝까지 읽는다. `sed -n '1,220p'`처럼 일부만 읽은 경우에는 전체 읽음으로 기록하지 않는다.
5. 실행 보고서에 rule별로 `적용`, `비적용`, `보류` 판단과 이유를 남긴다.
6. rule audit가 선행되지 않은 개발 태스크는 Done 처리하지 않는다. 사후 보정이면 `프로세스 위반과 정정`으로 기록한다.

기록 형식:

```markdown
| Rule file | Read scope | 판단 | 적용/비적용 이유 |
| --- | --- | --- | --- |
| `.claude/rules/common/workflow.md` | full | 적용 | report/update gate 기준 |
| `_domains/reward/rules/database.md` | full | 비적용 | 이번 태스크는 DB 변경 없음 |
```

### 플랜 구조
```markdown
## 목표
## 현재 상태 (AS-IS)
## 변경 후 상태 (TO-BE)
## 변경 범위 (파일 | 변경 내용 테이블)
## 영향성 (영향 대상 | 영향 내용 테이블)
## Breaking Changes
## 위험도 + 이유
## 주의사항
## 작업 단계
## 검증 방법
```

### 플랜 검토와 확정
- 플랜 검토 단계의 수정은 플랜 원문에 직접 반영한다

### 플랜 실행 이력
- 확정 전에는 `## 플랜 실행 이력` 섹션을 만들지 않는다
- 작업 범위가 확정되어 실행 착수하기 직전, 보고서 최상단에 `## 플랜 실행 이력`을 추가한다
- 이력은 스택 구조로 상단에 쌓는다 (최신이 위)
- 각 단계 완료, 검증, 보완, 커밋 시점마다 실제 실행 결과 기준으로 계속 갱신한다
- 한 작업이 QA 단계에서 끝나면 QA 완료 후 이력을 남긴다
- Operator 단계까지 진행되면 Operator 완료 후 이력을 남긴다
- Reviewer/QA/Operator 결과 또는 직접 검증 결과로 같은 작업 루프를 다시 돌려야 하면, 다음 루프를 시작하기 전에 현재 루프 결과와 보완 계획을 먼저 남긴다
- 흐름:
  1. 실행 착수 전 → `## 플랜 실행 이력` 섹션 생성, `### 실행 착수 전` 기록
  2. 파이프라인 마지막 단계 완료 → `### 작업 완료` 기록 (직접 실행/Reviewer/QA/Operator 중 실제 마지막 단계)
  3. 검증 결과 확인 → `### 검증 결과` 기록
  4. 보완 필요 발견 → 보고서에 `### 다음 루프 보완 계획` 기록 → 승인 필요한 등급이면 사용자 확인 후 Developer 재실행 또는 직접 보완
  5. 보완 작업 수행 → `### 보완 작업 완료` 스택 상단에 추가
  6. 추가 보완 필요 → 4-5 반복
  7. 커밋 시 → `### 커밋 기록` 기록

## 가이드라인 저장

### 핵심 원칙
실수/피드백 발생 시 반드시 규칙으로 저장한다. "기억", "저장", "메모리" 등 유사 문맥 요청 시 → 질의 후 저장한다. "인지했다"는 의미 없고, 기록해야 반복하지 않는다.

### 프로세스
1. 저장 요청 감지 → 저장할 내용 질의 (바로 저장하지 않음)
2. 일반화/특화 영역 판단 제시
3. 판단 근거와 함께 확인 요청
4. 승인 후 판단 기준에 따라 해당 규칙 문서에 저장 (일반화 → `common/` 또는 `language/`, 특화 → `project/`)

### 판단 기준
| 영역 | 위치 | 기준 |
|------|------|------|
| 일반화 | `common/` 또는 `language/` | 프로젝트 무관하게 적용 가능 |
| 특화 | `project/` | 특정 프로젝트 도메인에 종속 |

### 규칙 문서 원칙
- 성격이 다른 규칙끼리 통합하지 않는다
- 패턴명은 명확하게 (공식 용어 없으면 새 이름 생성)

### 기억 공유 프롬프트 형식
```
[파일경로] 파일에서 "[앞 섹션]" 섹션 끝과 "[뒤 섹션]" 섹션 시작 사이에
다음 내용을 추가해줘:
[추가할 내용 전체]
```

### 커밋 전 필수 확인
- 파이프라인 실행 보고서(`_domains/{domain}/reports/`)가 있으면 실행 이력을 갱신한다
- 실행 보고서를 변경 파일과 함께 스테이징한다
- 무관한 변경이 섞여 있으면 별도 커밋으로 분리한다

## 커밋 메시지

- 영어로 짧고 간결하게 작성
- 한 줄은 20자 이내
- 다른 내용은 별도 줄에 기록
- `_domains/{domain}/reports/`에 해당 작업의 실행 보고서가 있는지 확인하고, 맞으면 함께 커밋

```
refactor runtime iife
fix session storage
add bridge policy
```

## 작업 완료 후

### 필수 확인
1. 새로 작성한 함수/상수: JSDoc 주석 추가
2. 수정한 함수: 주석이 실제 동작과 일치하는지 확인
3. 외부 export: 사용 예시 포함

### 검증
- "확인했다" = 실제 대조 검증 (읽기만 했으면 "읽었습니다")
- 동작하는 코드/설정/의존성은 요청 없이 변경하지 않는다
