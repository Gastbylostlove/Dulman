# 마이그레이션

## 원칙: 원본 동작 보존

마이그레이션은 원본 코드의 관찰 가능한 동작을 재현하는 작업이다.

- 파라미터를 임의로 추가하거나 제거하지 않는다.
- 반환 타입, 에러 방식, 기본값, 부작용을 보존한다.
- "미래 대비" 옵션이나 새 기능을 함께 넣지 않는다.
- 동작 변경이 필요하면 마이그레이션과 분리해 별도 변경으로 다룬다.

## 마이그레이션 체크리스트

### 원본 분석

- [ ] 원본 호출부와 정의를 모두 확인한다.
- [ ] 입력 파라미터, 기본값, 선택 여부를 기록한다.
- [ ] 반환 타입, 에러 형식, 상태 변경, 외부 호출을 기록한다.
- [ ] 런타임에서 자동 주입되는 값이 있는지 확인한다.

### 대응 검증

- [ ] 원본 파라미터와 신규 코드의 파라미터가 1:1로 대응한다.
- [ ] API 경로, HTTP 메서드, 헤더, 버전, 권한 범위를 보존한다.
- [ ] 저장소 키, 이벤트명, 큐 이름, 토픽명처럼 외부 계약이 되는 문자열을 보존한다.
- [ ] 선택적 파라미터와 null/undefined 처리 방식이 원본과 일치한다.

### 사용처 검색

- [ ] 함수 직접 호출: `functionName(`
- [ ] 객체 속성 접근: `.propertyName`
- [ ] 구조분해 할당: `const { name } = value`
- [ ] 재export: `export { name } from`
- [ ] 타입 참조: `typeof name`, `ReturnType<typeof name>`
- [ ] 문자열 기반 참조: 설정, 라우트, 이벤트, 작업 이름

## @see 주석 형식

마이그레이션된 코드에는 원본 정의나 원본 동작을 추적할 수 있는 참조를 남긴다.

```typescript
// @see legacy/path/to/source.ts:120
const result = await newClient.call(input);
```

원본 코드를 길게 복사하지 않는다. 필요한 경우 핵심 차이와 보존해야 할 계약만 짧게 적는다.

## 완료 전 검증

플랜은 계약이다. 플랜의 모든 항목을 완료 전에 검증한다.

| 항목 | 사용처 | 반영 여부 | 확인 방법 |
| --- | --- | --- | --- |
| 파라미터 대응 | `src/example.ts` | 완료 | 호출부 검색 |

## 리팩토링 체크리스트

- [ ] 기존 코드베이스의 패턴과 스타일을 확인한다.
- [ ] 변경한 이름의 모든 참조를 업데이트한다.
- [ ] public API, export, import 경로가 일치하는지 확인한다.
- [ ] 타입 검사, 테스트, 빌드 중 가장 좁은 관련 검증을 실행한다.
- [ ] 문서나 주석이 실제 코드와 어긋나지 않는지 확인한다.
