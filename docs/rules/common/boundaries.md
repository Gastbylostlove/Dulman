# 경계와 계약

이 문서는 시스템의 **경계를 어디서 닫고, 무엇을 누가 소유하며, 정합성을 어떻게 지키는지**를 규정한다. 언어, 프레임워크, 프로젝트 종류와 무관하게 기본값으로 적용한다. 예외가 필요하면 근거와 대체 계약을 남긴다.

공통 실행 원칙은 `.claude/rules/common/principles.md`, 구체적 코드 원칙은 `.claude/rules/coding/principles.md`를 따른다.

## 소유

- **공유 계약은 단일 소유자를 둔다.** 같은 정책, 타입, 상수, 상태, 식별자, 데이터 형태가 여러 위치에서 독립 정의되면 위반이다.
- **공개 계약은 소비자가 아니라 소유자가 정의한다.** 소비자는 같은 데이터 형태를 재정의하지 않고 소유자가 제공한 계약을 사용한다.
- **계약값은 호출부에 흩뿌리지 않는다.** 외부와 맞물리는 문자열, 숫자, 상태, 포맷은 이름 붙은 계약으로 관리한다.

## 경계

- **정책을 가진 외부 경계는 소유자를 둔다.** retry, timeout, 직렬화, error mapping, consistency 보장이 필요한 경계는 호출부마다 직접 처리하지 않는다.
- **부작용은 경계에 모으고 핵심 판단과 분리한다.** 저장소, 네트워크, 시간, 랜덤성, 사용자 입력처럼 실행 환경에 의존하는 동작은 순수한 판단 로직과 섞지 않는다.
- **입력값 검증과 비즈니스 불변식은 구분한다.** 형식 검증은 entry boundary에서 먼저 수행하고, 상태 규칙은 core boundary에서도 보장한다.

## 정합성

- **성공으로 기록되는 side effect는 실제 성공과 일치해야 한다.** 실패하거나 미확정인 상태를 성공 이력, 완료 상태, 발급 상태처럼 남기면 위반이다.
- **계약은 happy path와 실패 조건을 함께 정의한다.** retry, timeout, 중복, race, rollback, partial success, error mapping을 정하지 않은 외부 연동은 미완성 계약으로 본다.
- **캐시는 성능 최적화가 아니라 일관성 계약으로 다룬다.** TTL, stale 허용 범위, invalidation 시점, 원본 fallback 중 무엇을 보장하는지 먼저 정하고 쓴다.

## 검증과 운영

- **테스트는 계약을 고정한다.** 입력 규칙, 외부 경계 실패, side effect 일관성, 상태 불변식을 계약 단위로 나눠 검증한다. 큰 시나리오 테스트 하나로 뭉뚱그리지 않는다.
- **운영 문서와 실행 명령은 실제 실행 가능한 상태를 유지한다.** 문서, 스크립트, 배포 설정, 생성 산출물 정책이 서로 어긋나면 프로젝트 계약이 깨진 것이다.

## 신규 프로젝트

- **구현 전에 boundary contract를 먼저 닫는다.** 누가 무엇을 소유하는지, 외부 경계를 어디서 닫는지, 실패를 어떻게 수렴하는지, 무엇을 테스트로 고정하는지를 먼저 합의한다.
