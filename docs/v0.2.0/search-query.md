## 1. 문서 개요

### 1.1 목적

현재 운영 중인 커플 채팅 앱에 **채팅방 검색 기능**을 추가하고, 이를 계기로 확인된 **통신/저장소 아키텍처의 비효율을 리팩토링**하기 위한 요구사항을 정의한다.

### 1.2 범위

- 신규 기능 1건 (채팅방 검색)
- 아키텍처 리팩토링 5건 (통신 방식, 백엔드 DB, 클라이언트 저장소, DB 스키마, 페이지네이션)
- 정책 명시화 1건 (서버 백업 정책 및 개인정보처리방침)
- 마이그레이션 계획

### 1.3 범위 외

- UI 전면 개편
- 인증/계정 시스템 변경
- 미디어 파일(S3) 저장 구조 변경

## 2. 현재 시스템 (As-Is)

### 2.1 기술 스택

| 계층                 | 현재                          |
| -------------------- | ----------------------------- |
| 클라이언트           | Flutter                       |
| 백엔드               | Node.js                       |
| DB                   | 로컬 PostgreSQL (상용 배포 X) |
| 미디어 저장          | AWS S3                        |
| 통신 방식            | HTTP 폴링 (3초 간격)          |
| 클라이언트 로컬 저장 | 없음                          |
| 검색 기능            | 없음                          |

### 2.2 확인된 이슈

#### 2.2.1 통신 비효율

- 3초마다 HTTP 요청 발생: 클라이언트 1대당 하루 약 28,800회 요청
- 새 메시지 유무와 무관하게 지속 통신
- 배터리 및 모바일 데이터 소모 큼

#### 2.2.2 백엔드 쿼리 비효율

- `listMessagesByChatId`가 DB에서 **전체 메시지를 로드한 뒤** JS 메모리에서 `.slice(0, 50)`로 자름
- DB 레벨 페이지네이션 미적용
- 메시지가 누적될수록 응답 시간 및 메모리 사용량 증가

#### 2.2.3 폴링 로직 버그

- 폴링 요청에 `after_message_id` 파라미터를 사용하지 않음
- 결과: 항상 가장 오래된 50건만 반복 수신
- 새 메시지가 51번째부터일 경우 폴링으로 수신 불가

#### 2.2.4 기능 부재

- 채팅방 내 메시지 검색 기능 없음
- 클라이언트 로컬 캐시 없음 → 오프라인 상태에서 이전 대화 조회 불가

### 2.3 현재 데이터 정책

- 서버(S3 및 로컬 PostgreSQL)에 대화 및 사진/영상이 이미 백업 저장되고 있음
- 저장 목적: 리벤지 포르노 등 범죄 악용 방지 및 법적 증거 자료 확보
- 개인정보처리방침에 백업 사실 명시 여부 및 정책 문서화 필요 (확인 필요)

### 2.4 현재 사용자 규모

- 사용자 미확보 (개발 단계)
- **의미**: 데이터 마이그레이션 리스크 낮음, 리팩토링 적기

## 3. 목표 시스템 (To-Be)

### 3.1 기술 스택 변경 요약

| 계층                 | 현재 (As-Is)    | 목표 (To-Be)                       |
| -------------------- | --------------- | ---------------------------------- |
| 클라이언트           | Flutter         | Flutter (유지)                     |
| 백엔드               | Node.js         | **Supabase (Node.js 폐기)**        |
| DB                   | 로컬 PostgreSQL | **Supabase PostgreSQL**            |
| 미디어 저장          | AWS S3          | AWS S3 (유지)                      |
| 통신 방식            | HTTP 폴링 3초   | **Supabase Realtime (WebSocket)**  |
| 클라이언트 로컬 저장 | 없음            | **SQLite (drift 패키지)**          |
| 검색 기능            | 없음            | **로컬 SQLite FTS5 + 형태소 토큰** |
| 페이지네이션         | 메모리 slice    | **커서 기반 (`after_message_id`)** |

### 3.2 데이터 아키텍처 원칙 (To-Be)

- **서버(Supabase) = 데이터 원본(source of truth)** — 법적 증거 자료 보관 겸함
- **클라이언트 SQLite = 로컬 캐시** — 빠른 UI, 오프라인 접근, 검색 인덱스
- **S3 = 미디어 저장소** — 메시지에는 참조 URL만 저장 (기존 유지)

## 4. 상세 요구사항

### 4.1 [신규] 채팅방 검색 기능

#### 4.1.1 UI 요구사항

- AppBar 우측 🔍 아이콘 탭 → AppBar가 검색창으로 전환 (X 버튼으로 닫기)
- 텍스트 입력 시 실시간 필터링
- 매칭된 메시지 목록 표시 (검색어 하이라이트 처리)
- 결과 탭 시 원본 메시지 위치로 스크롤

#### 4.1.2 검색 실행 위치

- **클라이언트 로컬 SQLite FTS5**에서 실행
- 서버 검색 API 미사용
- 근거: 네트워크 비용 최소화, 오프라인 지원, 서버 백업 비공개 정책과 일관성

#### 4.1.3 한국어 검색 파이프라인

**메시지 저장 시 (서버 측 처리)**

1. **정규화**: 소문자 변환, 공백 압축, 특수문자 정리, 유니코드 NFC, 흔들림 통일 (예: ㅋㅋㅋ → ㅋㅋ)
2. **형태소 분석**: Kiwi 등으로 어근 토큰 추출 (예: "사랑해" → "사랑")
3. **N-gram 인덱싱 (2~3글자)**: 신조어/은어 대응 (예: "사랑해" → "사랑", "랑해")
4. 원문은 `messages` 테이블, 검색용 토큰은 별도 컬럼 또는 `search_tokens` 테이블에 저장
5. 서버가 토큰까지 만들어 클라이언트로 전송 → 클라이언트는 SQLite FTS5 인덱스에 저장

**메시지 검색 시 (클라이언트 측 처리)**

1. 검색어 정규화
2. FTS5로 후보군 조회 (넉넉하게, 예: 100건)
3. Flutter 앱 로직으로 재정렬 및 필터링
   - 원문에 검색어 실제 포함 여부 재확인 (N-gram 노이즈 제거)
   - 정확 매칭 > 부분 매칭
   - 최근 메시지 우선
   - 등장 횟수 반영

#### 4.1.4 단계별 구현 방침

- **1단계 (MVP)**: 정규화 + N-gram만 구현. 형태소 분석기 없이 순수 SQLite로 검색 (약 90% 커버)
- **2단계**: 사용 로그 검토 후 검색 품질 부족 시 서버에 형태소 분석기 도입
- **3단계**: 노이즈 피드백 발생 시 앱 로직 보정 강화

---

### 4.2 [리팩토링] 통신 방식 (폴링 → WebSocket)

#### 4.2.1 변경 사항

- 3초 HTTP 폴링 완전 제거
- Supabase Realtime WebSocket 상시 연결로 대체
- 오프라인 상태에서 새 메시지 도착 시 Push 알림

#### 4.2.2 신규 메시지 흐름

**송신**

1. 사용자 메시지 입력 및 전송
2. 클라이언트 SQLite에 임시 저장 (status = pending)
3. UI 즉시 표시 (낙관적 업데이트)
4. WebSocket으로 서버 전송 (WSS + 인증 토큰)
5. 서버 검증 및 영속 저장 → message_id 생성
6. 송신자 ACK 회신 → SQLite status = sent
7. 수신자에게 실시간 push

**수신**

1. 수신자 온라인 시 WebSocket push, 오프라인 시 Push 알림
2. 클라이언트 SQLite 저장
3. UI 갱신
4. 사용자가 읽으면 read receipt 서버 전송
5. 서버가 송신자에게 read receipt 전달

#### 4.2.3 부수 효과

- **2.2.3 폴링 로직 버그(51번째 메시지 이후 미수신)는 폴링 제거로 자연 해결**

---

### 4.3 [리팩토링] 백엔드 저장소 (Node.js/로컬 PG → Supabase)

#### 4.3.1 변경 사항

- Node.js 백엔드 폐기
- 로컬 PostgreSQL 폐기
- Supabase(PostgreSQL 기반 BaaS) 도입

#### 4.3.2 도입 이유

- 자체 WebSocket 서버 구현 불필요 (Realtime 내장)
- Free tier로 초기 운영 가능 (500MB DB, 2GB 대역폭, 동시 연결 200)
- 인증, RLS, 스토리지, Edge Function 등 통합 제공

#### 4.3.3 필수 설정

- **Row Level Security (RLS)** 정책 필수: 커플 관계가 아닌 사용자의 데이터 접근 차단
- 인증 토큰 발급/검증 Supabase Auth 활용 여부 결정 필요

---

### 4.4 [리팩토링] 클라이언트 저장소 (신규 도입: SQLite)

#### 4.4.1 도입 목적

- 오프라인 상태 이전 대화 조회
- 빠른 UI 렌더링 (네트워크 왕복 제거)
- 검색 인덱스 저장소

#### 4.4.2 라이브러리 권장

- Flutter drift 패키지 (타입 안전, 마이그레이션 지원)

#### 4.4.3 동기화 전략

- 앱 최초 실행 시: 서버에서 전체 메시지 다운로드 → SQLite 저장
- 앱 재접속 시: 마지막 수신 message_id 이후 메시지만 fetch → SQLite 저장 (Delta sync)
- 실시간 수신: WebSocket push 시 SQLite에 즉시 저장

---

### 4.5 [리팩토링] DB 스키마 (1:1 하드코딩 → 1:다 대비)

#### 4.5.1 변경 방향

- MVP 기능은 1:1 유지
- **DB 스키마는 1:다 확장 가능한 구조로 설계**
- 근거: 카톡도 1:1에서 1:다로 확장한 전례, 미래 마이그레이션 비용 사전 절감

#### 4.5.2 스키마 구조 (제안)

```
chats
  - id
  - created_at
  - ...

chat_members
  - chat_id
  - user_id
  - role
  - joined_at

messages
  - id
  - chat_id
  - sender_id
  - content
  - search_tokens (또는 별도 테이블)
  - created_at
  - ...
```

#### 4.5.3 1:1 제약 방식

- **애플리케이션 레벨에서 `chat_members` 수 ≤ 2 강제**
- 1:다 확장 시 위 제약만 해제하면 됨

---

### 4.6 [리팩토링] 메시지 조회 페이지네이션

#### 4.6.1 변경 사항

- 전체 로드 후 메모리 slice 방식 폐기
- **커서 기반 페이지네이션 도입**: `GET /messages?after_id=1234&limit=50`
- DB 레벨에서 필요 건수만 조회

#### 4.6.2 부수 효과

- Supabase 이관 시 자연스럽게 수정 대상이 되므로, 리팩토링과 동시 진행 가능

---

### 4.7 [정책 명시화] 백업 및 개인정보 처리

#### 4.7.1 서버 백업 정책 (기존 유지 및 명문화)

- 대화 내용 및 사진/영상 서버 저장 유지
- 목적: 리벤지 포르노 등 범죄 악용 방지, 커플 간 분쟁 시 법적 증거 자료 활용
- 사용자에게 백업 조회 기능 미제공 (추후 검토)

#### 4.7.2 개인정보처리방침 업데이트 (필수)

- 개인정보보호법(PIPA) 준수
- 다음 사실 명시 및 사용자 동의:
  - 서버에 대화 및 사진/영상 백업 저장
  - 백업 목적 (범죄 방지 및 법적 증거)
  - 사용자는 서버 백업을 직접 조회할 수 없음
- 법 집행 기관 자료 요청 대응 프로세스 정의

#### 4.7.3 마케팅 문구 가이드라인

- **사용 금지**: "오직 둘만의 공간", "완전한 프라이버시", "우리만의 비밀", "E2EE" 등 오해 소지 표현
- **권장**: "안전한 커플 채팅, 문제 발생 시 증거로 활용 가능", "리벤지 포르노 방지" 등 서비스 목적을 명확히 하는 표현

## 5. 마이그레이션 계획

### 5.1 원칙

- 사용자 미확보 상태이므로 **데이터 마이그레이션 리스크 최소**
- 리팩토링 완료 후 기존 로컬 PostgreSQL 데이터 이전 여부 결정 (테스트 데이터면 폐기)
- 각 단계 완료 후 동작 검증

### 5.2 단계별 계획 (제안)

| 단계 | 작업                                                   | 예상 산출물                      |
| ---- | ------------------------------------------------------ | -------------------------------- |
| 1    | DB 스키마 재설계 (`chats`, `chat_members`, `messages`) | ERD, 마이그레이션 스크립트       |
| 2    | Supabase 프로젝트 셋업 + RLS 정책 정의                 | Supabase 프로젝트, RLS 정책 문서 |
| 3    | 클라이언트 SQLite 도입 (drift) + 스키마 정의           | drift 모델, 초기 sync 로직       |
| 4    | 기존 API를 Supabase 호출로 대체                        | 리팩토링된 Flutter 데이터 계층   |
| 5    | Supabase Realtime WebSocket 통합, 폴링 제거            | 실시간 수신 동작                 |
| 6    | 커서 기반 페이지네이션 적용                            | 개선된 메시지 로드               |
| 7    | 검색 기능 구현 (1단계: 정규화 + N-gram)                | 채팅방 검색 UI + FTS5 인덱스     |
| 8    | 개인정보처리방침 업데이트 및 필요 시 재동의 처리       | 방침 문서, 동의 UI               |
| 9    | 구 Node.js 백엔드 폐기                                 | 서버 인프라 정리                 |

### 5.3 리스크 및 대응

| 리스크                                       | 대응                                            |
| -------------------------------------------- | ----------------------------------------------- |
| Supabase Free tier 한계 초과                 | 사용량 모니터링 후 유료 전환 검토               |
| 형태소 분석기 서버 부담                      | 1단계는 N-gram만으로 시작, 필요 시 도입         |
| SQLite 마이그레이션 실패 (앱 업데이트 시)    | drift 마이그레이션 스크립트 표준 준수           |
| WebSocket 연결 안정성 (모바일 네트워크 변경) | Supabase SDK 재연결 로직 활용, 필요 시 fallback |

## 6. 미확정 사항 (다음 회의 안건)

| 항목                    | 내용                                                    | 우선순위 |
| ----------------------- | ------------------------------------------------------- | -------- |
