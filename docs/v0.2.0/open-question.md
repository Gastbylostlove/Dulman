### 1. 토큰을 리스트로 저장하나?

**아니요, 리스트가 아니라 공백으로 이어붙인 문자열로 저장합니다.**

이유: SQLite FTS5가 **공백 구분 텍스트**를 기본 입력으로 받거든요. 리스트를 그대로 넣을 수 없어요.

#### 실제 저장 형태

```sql
-- messages 테이블 (원문)
| id | content              | created_at |
|----|----------------------|------------|
| 1  | 사랑해 오늘도 많이 사랑했다 | ...        |

-- messages_fts (FTS5 검색용, 별도 가상 테이블)
| rowid | tokens                                         |
|-------|------------------------------------------------|
| 1     | 사랑 랑해 오늘 늘도 많이 사랑 랑했 했다             |
```

`tokens` 컬럼은 그냥 **긴 문자열 한 개**예요. 리스트 아님.

#### 왜 이렇게 하나

FTS5가 이 문자열을 받으면 **공백 기준으로 자동으로 쪼개서 인덱스**를 만들어요. 그래서 우리는 그냥 공백으로 이어붙여서 넣어주기만 하면 됩니다.

만약 진짜 리스트로 관리해야 하는 상황이면 별도 테이블을 쓰긴 해요:

```sql
-- 대안 (안 씀, 참고용)
| message_id | token |
|-----------|-------|
| 1         | 사랑   |
| 1         | 랑해   |
| 1         | 오늘   |
```

이건 검색은 되는데 **FTS5보다 느려요**. 그래서 FTS5 방식(공백 문자열)을 씁니다.

### 2. FTS5의 MATCH는 왜 빠른가?

이거 이해하려면 **인덱스**가 뭔지 알아야 해요. 한 문장으로 답하면:

**"미리 다 정리해둬서 빠른 겁니다."**

#### 인덱스가 없을 때 (`LIKE '%사랑%'`)

```
메시지 1: "사랑해 오늘도"     ← "사랑" 있나? 스캔... 있음!
메시지 2: "밥 먹자"           ← "사랑" 있나? 스캔... 없음
메시지 3: "잘 자"             ← "사랑" 있나? 스캔... 없음
메시지 4: "사랑스러운 강아지"  ← "사랑" 있나? 스캔... 있음!
...
메시지 10만: "..."           ← 스캔... 없음
```

**10만 개 메시지 다 하나씩 열어봄.** 느림.

#### 인덱스가 있을 때 (FTS5 MATCH)

FTS5는 저장할 때 **역방향 인덱스**를 미리 만들어놔요. 책 뒤 "찾아보기"랑 완전 같은 개념:

```
FTS5가 미리 만들어둔 인덱스 (책 뒤 찾아보기 같은 구조)

토큰       →  이 토큰이 있는 메시지 id들
"사랑"    →  [1, 4, 27, 103, 5821, ...]
"오늘"    →  [1, 8, 44, ...]
"밥"      →  [2, 15, 88, ...]
"잘"      →  [3, 19, 66, ...]
"강아지"   →  [4, 205, ...]
```

`MATCH '사랑'` 하면:

1. 인덱스에서 "사랑" 항목 찾음 (즉시)
2. `[1, 4, 27, 103, 5821, ...]` 반환 (즉시)

**10만 개 스캔 안 함.** 인덱스 한 번만 조회.

#### 비유

- **LIKE 검색**: 책에서 "사랑"이란 단어 찾는데 1페이지부터 마지막까지 다 읽음
- **FTS5 MATCH**: 책 뒤 "찾아보기"에서 "사랑" 항목 찾아서 "127, 342페이지" 바로 감

#### 속도 차이

10만 건 기준:

- LIKE: 수 초
- FTS5 MATCH: 수 ms (1000배 빠름)

#### 대가

공짜는 아니에요. 인덱스를 저장해야 하니까 **저장 공간이 원문의 2~3배** 필요해요. 근데 요즘 폰 용량 넉넉하니까 문제 안 됨.

### 3. Supabase에서 토큰화가 병목이 되지 않나?

**성진님이 정말 잘 짚으신 부분입니다. 실제로 될 수 있어요.**

#### 병목 가능성 계산

Kiwi 형태소 분석 속도: 대략 **한 메시지당 1~5ms**

- 초당 100건 메시지: 100~500ms 처리 → 여유
- 초당 1000건 메시지: 1~5초 처리 → **위험**
- 초당 10000건 메시지: 10~50초 → **완전 병목**

#### 우리 앱 규모에서는?

MVP 초기 (커플 몇 쌍):

- 하루 몇 백 건 메시지
- 초당 0.01~0.1건 수준
- **병목 발생 확률: 0%**

성장 후 (커플 1만 쌍, 하루 100만 건):

- 초당 평균 11.5건, 피크 시간 초당 100건 정도
- **여전히 여유** (1000건 이상 되어야 위험)

#### 그럼 언제 문제가 되냐

- 커플 100만 쌍 → 실질적으로 카톡 규모
- 이때는 이미 서버 여러 대로 늘리고 있을 시점
- **지금 걱정할 문제 아님**

#### 그래도 대비하려면

1. **캐싱**: 같은 메시지 여러 번 토큰화 안 하도록
2. **분산 처리**: 서버 여러 대에서 나눠서 토큰화
3. **경량화**: Kiwi 대신 더 가벼운 토크나이저 부분 도입

이건 나중에 문제 생기면 그때 하시면 되고, MVP에서는 신경 안 써도 됩니다.

### 4. Realtime 전달과 토큰화를 분리해서 백그라운드로?

**이게 오늘 나온 질문 중 최고예요. 실무에서 진짜 중요한 판단이에요.**

두 가지 선택지가 있어요:

#### 방식 A: 동기 처리 (토큰화 완료 후 전달)

```
1. 메시지 도착
2. Kiwi 토큰화 (5ms)
3. DB에 원문 + 토큰 저장 (10ms)
4. Realtime으로 상대방에게 push
5. 총 지연: 15ms
```

**장점**: 단순함, 항상 토큰 완비된 상태

**단점**: 토큰화가 느리면 메시지 전달도 느려짐

#### 방식 B: 비동기 처리 (전달 먼저, 토큰화 백그라운드)

```
1. 메시지 도착
2. DB에 원문만 저장 (5ms)
3. Realtime으로 상대방에게 push  ← 상대는 즉시 받음
4. [백그라운드] Kiwi 토큰화 시작 (별도 프로세스)
5. 토큰화 완료되면 DB에 토큰 컬럼 업데이트
6. 추가 Realtime 이벤트로 토큰만 다시 push
```

**장점**: 메시지 전달 빠름 (토큰화 대기 안 함)

**단점**: 복잡함, 토큰 도착 전에 사용자가 검색하면 미완성 인덱스

#### 우리 앱에는 뭐가 맞나?

**MVP에서는 방식 A(동기 처리)로 충분해요.**

이유:

1. 메시지 하나당 15ms는 **사용자가 느낄 수 없는 속도**
2. 사용자 규모 작아서 서버 부하 없음
3. 복잡도 낮음 → 개발 빠름
4. 나중에 문제 되면 그때 B로 전환

**언제 방식 B로 전환?**

- 메시지 전달 지연이 100ms 넘어갈 때
- 서버 CPU가 토큰화로 지속 부하 받을 때
- 초당 1000건 이상 처리해야 할 때
